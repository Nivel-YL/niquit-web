# AI-редактор блога — сам механизм автоматизации

BLOG_TOPIC_BACKLOG.md и CONTENT_VOICE_GUIDE.md — это правила, которые этот механизм должен соблюдать, а не сам механизм. Ниже — что реально нужно построить, чтобы система запускала себя сама.

## Общая схема

Три независимых этапа, не один сплошной процесс:

1. **Генерация** — темп не ограничен искусственно. Черновик, лежащий неопубликованным в репозитории, не создаёт риска независимо от скорости появления — значит нет причин тормозить этот этап под темп публикации.
2. **Проверка** — раз в 2-4 недели, пачкой, вручную.
3. **Публикация** — через очередь с разнесёнными датами, полностью автоматически после одобрения (см. ниже) — именно здесь важен равномерный темп, не на этапе генерации.

---

## 1. Генерация — расписание не задаёт темп, только напоминает системе о себе

```yaml
# .github/workflows/blog-editor.yml
on:
  schedule:
    - cron: '0 9 * * 1'     # раз в неделю — достаточно; темп публикации отдельный
  workflow_dispatch:
    inputs:
      batch_size:
        default: '3'
```

Берёт следующую невзятую тему из backlog — один раз исследует универсальные факты через веб-поиск (см. раздел 3 — не по одному разу на язык) — для каждого из 5 языков пишет статью на основе общих фактов + своего языкового довеска, если есть — сохраняет как markdown с `draft: true` — помечает тему использованной — коммитит. Может отработать несколько тем за один запуск (`BATCH_SIZE`), если есть смысл не растягивать — эта скорость больше не связана с тем, как часто что-то появляется на живом сайте.

---

## 2. Выбор темы

Скрипт читает блок `<!--BACKLOG ... BACKLOG-->` из `BLOG_TOPIC_BACKLOG.md`, находит первую тему со статусом `pending`, использует её как основу для всех 5 языков в этом запуске. После успешной генерации меняет статус на `in_progress`, а после того как все 5 черновиков реально сохранены — на `drafted`. Статус `published` для конкретного языка проставляется отдельно, когда снимается `draft: true` (см. ниже, это можно связать одним небольшим скриптом).

---

## 3. Исследование и написание — переработано целиком, не патч

> Этот раздел полностью заменён документом `docs/CONTENT_RELIABILITY_SPEC.md`.
> Ниже — краткое резюме того, что реализовано.

### Главное структурное изменение: исследование один раз на тему, не 5 раз на язык

**Было**: для каждого из 5 языков — отдельный, независимый вызов с поиском.
**Проблема**: 5 независимых попыток могут найти 5 разных цифр для одного и того же реального явления — риск несостыковки между языковыми версиями, и переплата за поиск, который можно было сделать один раз.

**Новая схема** (реализована в `scripts/blog_editor.py`):

1. **Один общий вызов с поиском** — собирает универсальные факты по теме: статистику, механизмы, актуальные данные, каждый факт с источником и датой.
2. **Отдельная проверка**: нужен ли для этой темы ещё и страно-специфичный контекст (юридический статус, культурный угол вроде DiGA в Германии) — если да, точечный дополнительный поиск именно на нужном языке, не на все 5 сразу.
3. **5 вызовов на написание** — каждый получает один и тот же общий набор фактов + свой языко-специфичный довесок, если он есть, и пишет независимо, нативно, каждый на своём языке, с полным текстом `CONTENT_VOICE_GUIDE.md` на входе.

Побочный эффект: это ещё и дешевле, чем было — один исследовательский вызов на тему вместо пяти.

### Дисциплина проверки фактов — обязательные правила для исследовательского вызова

- Любая конкретная цифра должна быть подтверждена минимум двумя независимыми источниками. Если источники расходятся — использовать диапазон ("от X до Y"), не одно правдоподобное число.
- Каждый факт передаётся на этап написания вместе с источником и датой публикации источника.
- Источник моложе 3 лет — предпочтительно, особенно для юридических и регуляторных фактов.
- Иерархия качества источников: официальные органы здравоохранения и государства (ВОЗ, Минздрав и аналоги) → рецензируемые исследования → авторитетные новостные издания → всё остальное.
- Юридические факты — отдельная категория: явно указывать "по состоянию на [дата источника]", не как неизменную истину.

### Механические правила — проверяются кодом, не доверяются памяти модели

Простой скрипт (без вызова ИИ, без доп. стоимости), который проверяет готовый текст сразу после этапа написания:

- есть ли символ "—" где угодно в тексте;
- встречается ли любая фраза из фиксированного списка запрещённых оборотов (`BANNED_PHRASES` в `blog_editor.py`).

Если проверка не пройдена — черновик сохраняется с явным флагом `style_check: "failed — ..."` в frontmatter, чтобы это было видно сразу при просмотре, а не потерялось в тексте. Это отличается от проверки фактов ниже тем, что тут не нужно суждение — правило либо соблюдено, либо нет.

### Факты и цифры — разовой настройки не будет никогда, и это нормально

В отличие от механических правил, точность конкретной цифры в новой статье на новую тему — это не то, что можно проверить раз и навсегда одним правилом. Аудит по фактам с источниками (ниже) не устраняет риск полностью ни при какой настройке — он делает периодическую проверку быстрой и предметной, а не превращает её в ненужную. Это не недоработка системы, а естественное свойство любой работы с новыми фактами, человеческой или ИИ.

### Правило "не медицинский совет" — общее, не только для отдельных тем

Этап написания может описывать механизмы, статистику, исследования, но не формулировать "поэтому тебе стоит/не стоит X" — это работа врача, не блога, независимо от того, в какой теме это всплывает.

### Внутренние ссылки — фактически заработавшая версия

Этап написания получает на входе список уже опубликованных (не просто сгенерированных) статей — заголовок и слаг — иначе инструкция "ссылайся на предыдущие статьи" не может выполниться. Список обновляется автоматически при каждой публикации.

### Проверка на дублирование внутри кластера

В одном кластере несколько тем подряд — этап написания получает явное предупреждение не повторять те же вводные объяснения, которые уже разбирались в другой статье того же кластера.

### Новый слой: автоматический отчёт по фактам для проверки

Отдельный, третий, короткий вызов (без поиска, дешёвый, Haiku) после того как статья написана:

- Извлекает из готового текста каждую конкретную цифру/статистику/юридический факт
- Сопоставляет с тем, что было передано с исследовательского этапа
- Если что-то в финальном тексте не соответствует найденным фактам — явно помечается

Результат: короткий отчёт рядом с каждым черновиком (`docs/fact-audits/{slug}/{lang}.md`): список конкретных фактов, использованных в статье, с источником у каждого. Проверяющий смотрит не сплошной текст в поисках подвоха, а именно этот компактный список — отклонение видно сразу.

---

## 4. Сохранение и коммит (генерация)

Каждый черновик — `src/content/blog/{lang}/{slug}.md`. Обновлённый `BLOG_TOPIC_BACKLOG.md`. Файлы аудита — `docs/fact-audits/{slug}/{lang}.md`. Один коммит на весь запуск, автоматический пуш в рабочую ветку — дальше это просто обычные файлы в репозитории, ничего специального Netlify не нужно (сборка их не публикует, пока `draft: true`).

---

## 5. Публикация — очередь с разнесёнными датами, не сразу при одобрении

Ключевое отличие от более ранней версии архитектуры: одобрение статьи (снятие `draft: true`) не публикует её немедленно, а ставит в очередь на будущую дату.

**При одобрении** — скрипт (`scripts/publisher.py`) переключает `draft: true` → `draft: false` и обновляет `publishDate` до следующего свободного слота в очереди: находит последнюю уже занятую дату среди одобренных, но ещё не наступивших статей, прибавляет интервал (+3-4 дня под темп 2 раза в неделю), ставит новую дату следующей по очереди. Так несколько одобренных за один раз статей не публикуются одним днём, а автоматически растягиваются во времени.

**Отдельный cron** (`.github/workflows/publisher.yml`, вторник + пятница 08:00 UTC) делает Netlify rebuild через git push — при каждой такой пересборке Astro включает в финальную сборку только статьи, где одновременно `draft: false` и `publishDate` уже наступил.

Итог: редактор один раз говорит "эти одобрю" — а дальше публикация растягивается сама, равномерно, без единого дополнительного действия с его стороны в момент, когда каждая конкретная статья реально появляется на сайте.

## Что нужно настроить один раз (не код, а секреты доступа)

- `ANTHROPIC_API_KEY1` — ключ Anthropic API, в секретах репозитория GitHub.
- `GITHUB_TOKEN` — встроенный токен с правами записи (`permissions: contents: write` в workflow).
- Netlify auto-deploy работает через git push в master (публичный репозиторий, ограничений по contributors нет).

## Стоимость

Sonnet 5, вводная цена $2/$10 за миллион токенов вход/выход (действует до 31 августа 2026, далее $3/$15). Haiku 4.5 — $1/$5.

На один запуск (3 темы × 5 языков):
- 3 × (1 shared research call Haiku + 5 write calls Sonnet 5 + 5 audit calls Haiku) = ~3-7 поисковых вызовов + 15 Sonnet + 18 Haiku
- С запасом: $0.5-0.8 за запуск (3 темы)

Итого: примерно **$2-3 в месяц** при одном запуске в неделю. Не бюджет, о котором стоит переживать — но стоит сверить с реальным расходом после первых 2-3 запусков.

## Обложки статей — программные карточки, не AI-иллюстрации

Отдельный шаг, полностью автоматический, без AI-генерации изображений и без стоковых фото (авторские права — реальный риск). Карточка — SVG, генерируется кодом при создании черновика: фирменный navy-градиент фона + крупная полупрозрачная иконка кластера, с небольшим детерминированным смещением/поворотом на основе id темы (чтобы статьи одного кластера не выглядели identично), не случайным — тот же id всегда даёт тот же результат.

Иконки по кластеру:
- A (пачи/снюс): 🤝
- B (вейп): 💨
- C (сигареты): 🚬
- D (мульти-источник): 🎯
- E (психология/метод): 🧠
- F (здоровье): ❤️
- G (практика): ⚡
- H (мифы): 🔮

Сохраняется как `public/images/blog/{slug}.svg`, путь прописывается в `heroImage` черновика. Ноль дополнительной стоимости, ноль риска несоответствия бренду, ничего для проверки человеком.

---

## Готово, когда

Ручной запуск генерации производит корректные markdown-черновики с уже готовой SVG-обложкой у каждого, обновляет статус темы в backlog, закоммичено одним коммитом без чего-либо участия. Ни один черновик не проходит дальше с тире или запрещённой фразой — код проверяет это до сохранения. Исследование по теме выполняется один раз, а не пять — каждая цифра в готовых статьях имеет источник и дату, видимые в отдельном отчёте рядом с черновиком. Ссылки между статьями кластера реально появляются в тексте. Отдельно: одобрение статьи автоматически ставит её в очередь с корректной будущей датой, cron открывает статьи строго по датам их очереди, без ручного вмешательства в момент публикации каждой конкретной статьи.
